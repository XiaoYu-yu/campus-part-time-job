# 内测用户试用说明与账号发放边界

## 目标

本说明用于 owner-controlled 内测，不用于公开正式上线。

当前项目状态是：

```text
本地 / 内测型试运营可用
但不是完整产品级正式上线
```

## 参与范围

建议第一轮只给少量可信测试者使用：

1. owner 本人。
2. 1 到 3 名熟悉项目背景的同学或朋友。
3. 不建议开放给不受控公网用户。
4. 不建议用于真实交易、真实支付、真实打款或真实履约承诺。

## 账号发放边界

账号分三类：

1. admin：用于管理后台。
2. customer：用于用户端发布代送、查看结果、确认完成。
3. parttime：用于兼职端登录、接单、取餐、送达、异常上报。

账号和密码发放规则：

1. 不把账号密码写入公开仓库。
2. 不把账号密码写入截图、录屏或 README。
3. 由 owner 私下发放给测试者。
4. 每个测试者只拿自己需要的角色账号。
5. 测试结束后如有必要，重置测试账号或重置 H2 / MySQL 数据。

## 当前建议测试链路

### customer 用户端

可以测试：

1. 登录用户端。
2. 查看校园代送入口。
3. 发布校园代送单。
4. 模拟支付。
5. 查看订单结果。
6. 查看 completed 结果回看。
7. 入驻兼职资料链路。
8. 申请兼职 token。

不要测试：

1. 真实支付。
2. 真实退款。
3. 真实银行卡、微信、支付宝等资金链路。
4. 大量并发下单。
5. 真实地址隐私数据。

### parttime 兼职端

可以测试：

1. 登录兼职端。
2. 查看个人资料和审核状态。
3. 查看可接单列表。
4. 接单。
5. 查看订单详情。
6. 取餐。
7. 送达。
8. 异常上报。
9. completed 回读。

不要测试：

1. 真实地图导航。
2. 真实定位轨迹。
3. 真实骑手调度。
4. 批量刷单或压力测试。

### admin 管理端

可以测试：

1. 运营总览。
2. 运营人员管理。
3. campus settlement 只读运营页。
4. after-sale 执行只读页。
5. courier 运营 / 位置预览页。
6. exception 历史与 resolve 最小处理页。
7. settlement 批次、对账差异和操作审计相关只读页。
8. 旧兼容入口的“兼容模块”展示。

不要测试：

1. 删除旧外卖模块。
2. 修改 bridge。
3. 修改鉴权。
4. 真实财务结算。
5. 大规模后台数据导入。

## 反馈记录格式

建议每条反馈按以下格式记录：

```text
反馈人：
角色：admin / customer / parttime
页面：
操作步骤：
预期结果：
实际结果：
是否可复现：
截图或录屏：
严重程度：阻塞 / 主要 / 次要 / 建议
```

严重程度建议：

1. 阻塞：无法登录、无法进入主链路、服务不可用。
2. 主要：核心动作失败，例如下单、接单、取餐、送达、确认。
3. 次要：页面样式错位、文案不清晰、空态不友好。
4. 建议：体验优化或后续产品建议。

## 测试前检查

测试前 owner 应确认：

1. 服务器 frontend 可访问。
2. `GET /api/campus/public/health` 返回 `UP`。
3. `server-post-deploy-check.ps1` 通过。
4. `server-ops-health.ps1` 无磁盘压力。
5. 账号已私下发放。
6. 已说明当前为模拟资金链路。
7. 已说明当前没有 HTTPS / 域名 / 证书。

## 出问题时如何处理

1. 先保存截图或录屏。
2. 记录角色、页面、操作步骤和时间。
3. owner 先跑 `server-post-deploy-check.ps1` 判断是否服务器整体异常。
4. 如果 remote smoke 通过，则优先判断为单页面或单接口问题。
5. 如果 remote smoke 失败，则先暂停继续测试，排查服务器。

## 当前明确声明

1. 真实支付。
2. 真实退款。
3. 真实打款。
