# 内测反馈记录与分级规则

## 目标

内测反馈必须能落到可判断、可复现、可排优先级的记录上，避免只留下“感觉不好用”“页面有问题”这类不可执行反馈。

本规则用于 owner-controlled 内测阶段。

## 反馈记录模板

每条反馈按以下格式记录：

```text
反馈编号：
反馈人：
反馈时间：
角色：admin / customer / parttime
设备：Web / Android 用户端 / Android 兼职端
页面入口：
操作步骤：
预期结果：
实际结果：
是否可复现：是 / 否 / 未确认
截图或录屏：
接口或错误信息：
严重程度：阻塞 / 主要 / 次要 / 建议
处理结论：待确认 / 进入修复 / 后续规划 / 不处理
负责人：
备注：
```

## 严重程度定义

### 阻塞

满足任一条件即为阻塞：

1. 无法登录。
2. 服务不可访问。
3. customer / parttime / admin 任一主入口打不开。
4. 核心主链路不可继续，例如无法发布代送单、无法接单、无法送达、无法确认。
5. 数据明显错乱，影响后续测试判断。

处理策略：

1. 暂停相关链路继续测试。
2. 先跑 `server-post-deploy-check.ps1`。
3. 若远端 smoke 失败，先修服务器或接口。
4. 若远端 smoke 通过，定位单页面或单接口问题。

### 主要

满足任一条件即为主要：

1. 核心动作偶发失败但可绕过。
2. 重要状态展示错误。
3. 页面按钮或表单会误导用户。
4. admin 运营页关键字段缺失或展示错误。
5. Android WebView 和 Web 表现不一致且影响主链路。

处理策略：

1. 进入近期修复候选。
2. 修复前必须先明确复现步骤。
3. 不为模糊反馈直接改代码。

### 次要

满足任一条件即为次要：

1. 视觉不够美观但不影响操作。
2. 文案不够准确但不误导关键动作。
3. 空态、loading、错误态不够清楚。
4. 非核心页面样式轻微错位。

处理策略：

1. 进入维护候选。
2. 累积到一组后统一修，不单独开大改。

### 建议

满足任一条件即为建议：

1. 希望增加新功能。
2. 希望改交互流程。
3. 希望接真实支付、真实地图导航、消息通知等后续能力。
4. 希望重做视觉风格。

处理策略：

1. 不直接进入当前修复。
2. 归档到后续产品规划。
3. 必须先评估是否触碰 bridge、支付、地图、隐私或上线边界。

## 进入修复的条件

只有同时满足以下条件才进入修复：

1. 有明确页面或接口。
2. 有复现步骤。
3. 有预期结果和实际结果。
4. 严重程度为阻塞或主要，或次要问题已成组出现。
5. 修复范围不违反当前冻结线。

当前冻结线：

1. bridge 主线冻结。
2. 展示 polish 线维护态。
3. 媒体线收住。
4. 旧外卖模块不直接删除。

## 不进入当前修复的反馈

以下反馈不进入当前修复：

1. “想做正式支付”。
2. “想接真实打款”。
3. “想改成正式地图导航”。
4. “想开放公众注册”。
5. “想删除旧外卖模块”。
6. 没有复现步骤的模糊体验反馈。
7. 只影响非当前内测范围的建议。

这些反馈应进入后续规划或专项设计，不应打乱当前内测稳定线。

## 建议文件归档方式

内测期间可在本地维护一个不提交的反馈表：

```text
tmp/internal-trial-feedback.md
```

如果反馈已经脱敏且需要纳入项目记录，再整理成正式日志。

不要提交：

1. 测试者真实手机号。
2. 真实姓名。
3. 真实地址。
4. 账号密码。
5. 服务器 IP 或密钥。
6. 未脱敏截图或录屏。

## 每轮修复前检查

每次从反馈进入修复前，先回答：

1. 这是不是当前内测范围？
2. 是否有复现步骤？
3. 是否影响主链路？
4. 是否触碰 bridge / 鉴权 / token / 支付 / 地图 / 旧模块删除？
5. 是否可以单提交回滚？
6. 是否需要先补方案而不是直接写代码？

如果答案不清楚，不应直接修。
