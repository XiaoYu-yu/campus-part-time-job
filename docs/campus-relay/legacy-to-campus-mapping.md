# 旧模块到校园代送模块映射

> 状态：规划中，未实现
>
> 本文只说明旧模块后续如何处理，不表示本轮已完成替换。本轮不产生运行时行为变化。

## 1. 处理原则

- `复用`：基础设施或角色语义可直接沿用，仅在后续轮次补字段或改文案。
- `隐藏`：功能不再面向最终校园代送场景展示，但短期保留代码和表，避免一次性大拆。
- `替换`：旧模块语义与校园代送核心业务不一致，后续以新 `campus_*` 领域模型替代。
- `延后删除`：当前先保留以支持过渡演示或回归对照，在第 09 轮统一收口。

## 7. 当前结论

- 旧外卖系统不会在一轮内被硬删除，而是以“并行新增校园代送域”的方式逐步替换。
- 复用的重点是管理员、普通用户登录基础、上传、安全、分页、工程结构。
- 替换的重点是菜品、套餐、购物车、收货地址、店铺营业、外卖订单语义和用户端页面。
- 当前用户 Web 页面和旧表都会保留到第 09 轮统一清理，避免在第 02 轮就因为删旧模块导致大面积联调失败。
