# show_shop1 项目检查说明

检查时间：2026-04-03

## 1. 项目概览

### 1.1 项目是什么

`show_shop1` 是一个前后端分离的外卖/餐饮管理系统，目标上包含两部分：

- 管理后台：员工、分类、菜品、套餐、订单、统计等管理能力
- 用户端：首页、分类、菜品详情、购物车、下单、订单、个人中心等消费端能力

### 1.2 技术栈

- 前端：Vue 3、Vite、Element Plus、Pinia、Vue Router、ECharts
- 后端：Spring Boot 3、MyBatis、MySQL、JWT、Maven

### 1.3 目录结构

- `frontend/`：前端项目
- `backend/`：后端项目
- `docs/`：接口与数据库设计文档
- `README.md`：根目录项目说明

用户端页面文件很多，但大部分仍处于 UI 展示、mock 数据驱动或硬编码状态，未形成真实业务闭环。当前用户端页面包括：

- `frontend/src/views/user/Orders.vue`
- `frontend/src/views/user/Profile.vue`

这些页面普遍存在以下现象：

- 直接依赖 `useMockStore`
- 页面内硬编码商品、订单、用户信息
- 大量使用临时图片 URL
- 关键动作仍是 `console.log(...)` 或提示“开发中”

### 2.3 权限与联调状态

- 管理后台登录接口存在，前端登录页也接入了员工登录
- 但前端路由没有登录守卫
- 后端拦截器几乎拦截全部接口，只放行员工登录与静态上传路径
- 用户端没有独立的认证、购物车、地址、下单等后端接口支撑

结论：项目当前更接近“管理后台部分完成 + 用户端原型页面齐全但未真正联调”。

- 当前 `frontend/package.json` 只有 `dev`、`build`、`preview` 脚本，没有 `lint` 或 `test`

### 3.2 后端检查

执行目录：`backend`

执行命令：

```powershell
..\apache-maven-3.9.14\bin\mvn.cmd test
..\apache-maven-3.9.14\bin\mvn.cmd -q -DskipTests compile
```

结果：

- 两个命令都失败
- 失败原因一致：Lombok 依赖无法解析

直接原因：

- `backend/pom.xml` 使用了 `org.projectlombok:lombok:edge-SNAPSHOT`
- Maven 在 `https://projectlombok.org/edges` 无法解析该依赖

影响：

- 后端当前无法稳定编译
- 测试无法执行
- 本地联调和交付都被阻塞

### 3.3 测试覆盖检查

后端测试目录仅发现一个空测试文件：

- `backend/src/test/java/com/cangqiong/takeaway/TakeawayApplicationTests.java`

当前现状：

- 后端只有 `contextLoads()` 空测试
- 前端没有自动化测试
- 前端也没有 lint 脚本

## 4. 缺陷清单

### 4.1 高优先级问题

#### 4.1.1 后端不可构建

- 现象：`backend/pom.xml` 中的 Lombok 版本为 `edge-SNAPSHOT`
- 影响：后端无法编译，测试无法执行，阻塞联调与交付
- 位置：`backend/pom.xml`
- 建议：改为 Maven Central 中可稳定获取的正式版 Lombok，并清理失败缓存后重新编译

#### 4.1.2 数据库表名与 Mapper 不一致

- 现象：初始化脚本创建的是 ``order`` 表，但 `OrderMapper`、`StatisticsMapper` 查询和写入的是 `orders`
- 影响：即使后端编译通过，订单与统计相关接口也会直接因表不存在而报错
- 位置：
  - `backend/db/init.sql`
  - `backend/src/main/java/com/cangqiong/takeaway/mapper/OrderMapper.java`
  - `backend/src/main/java/com/cangqiong/takeaway/mapper/StatisticsMapper.java`
- 建议：统一订单表命名，优先明确数据库真实表结构，再同步修正 SQL 与实体注释

#### 4.1.3 订单接口契约不一致

- 现象：前端传的是 `orderNo`、`customerName`、`beginTime`、`endTime`、中文状态字符串；后端只接收 `status/startDate/endDate`，并且 `status` 是 `Integer`
- 影响：订单筛选、状态更新、列表查询会出现参数失配，页面行为与接口语义不一致
- 位置：
  - `frontend/src/api/order.js`
  - `frontend/src/views/Order.vue`
  - `backend/src/main/java/com/cangqiong/takeaway/query/OrderQuery.java`
  - `backend/src/main/java/com/cangqiong/takeaway/controller/OrderController.java`
- 建议：先统一状态枚举和查询字段命名，再统一前后端请求与响应结构

#### 4.1.4 统计接口契约不一致

- 现象：前端请求 `/statistics/dashboard`、`/statistics/sales`、`/statistics/popular-dishes`、`/statistics/users`，后端只提供 `/statistics/turnover`、`/statistics/orders`、`/statistics/dishes`
- 影响：仪表盘和统计页依赖的真实接口并不存在，页面只能报错或展示空数据
- 位置：
  - `frontend/src/api/statistics.js`
  - `frontend/src/views/Dashboard.vue`
  - `frontend/src/views/Statistics.vue`
  - `backend/src/main/java/com/cangqiong/takeaway/controller/StatisticsController.java`
- 建议：二选一统一方向
  - 要么按前端页面补齐后端统计聚合接口
  - 要么重构前端，改用后端现有统计接口组合页面

#### 4.1.5 用户端未真正联调

- 现象：用户端页面和 `UserLayout`、`ShopStatus` 仍大量依赖 mock/hardcode
- 影响：用户端目前不是可用业务系统，而是高保真静态/半静态原型
- 位置：
  - `frontend/src/layout/UserLayout.vue`
  - `frontend/src/views/ShopStatus.vue`
  - `frontend/src/views/user/*.vue`
  - `frontend/src/stores/mock.js`
- 建议：先补用户端后端接口，再逐页去 mock 化

#### 4.1.6 权限链路不完整

- 现象：前端路由没有登录守卫；后端 JWT 拦截器几乎拦截全部接口，只放行员工登录
- 影响：
  - 管理后台未登录也能直接访问页面路由
  - 用户端没有独立认证链路
  - 未来即使接入真实接口，也会在鉴权设计上卡住
- 位置：
  - `frontend/src/router/index.js`
  - `backend/src/main/java/com/cangqiong/takeaway/config/WebMvcConfig.java`
- 建议：拆分管理端与用户端的认证策略，并补前端路由守卫与用户态管理

### 4.2 中优先级问题

#### 4.2.1 根 README 描述超前

- 现象：根 `README.md` 将用户端、截图、测试账号和整体业务闭环描述得较完整
- 影响：对外阅读时容易误判项目完成度
- 位置：`README.md`
- 建议：按“已完成 / 部分完成 / 未完成”重写说明

#### 4.2.2 README 截图全部失效

- 现象：`README.md` 引用了 `docs/screenshots/*`，但仓库中不存在该目录
- 影响：文档展示失真，影响项目说明质量
- 位置：
  - `README.md`
  - `docs/screenshots/` 不存在
- 建议：补齐截图资源，或删除无效截图引用

#### 4.2.3 `docs/README.md` 与实际目录不一致

- 现象：文档里写有 `api/`、`database/`、`design/` 等目录，但实际 `docs/` 下只有三个 Markdown 文件
- 影响：文档结构说明不准确
- 位置：`docs/README.md`
- 建议：按实际文件重写文档索引

#### 4.2.4 `frontend/README.md` 仍是默认模板

- 现象：前端 README 还是 Vite 初始化模板
- 影响：前端子项目缺少真实运行说明、环境变量说明和页面结构说明
- 位置：`frontend/README.md`
- 建议：替换为项目专用 README

#### 4.2.5 生产环境时区配置格式异常

- 现象：`application-prod.properties` 中写的是 `serverTimezone=Asia Shanghai`
- 影响：MySQL 连接参数可能解析异常，生产环境存在配置风险
- 位置：`backend/src/main/resources/application-prod.properties`
- 建议：改为合法时区格式，例如 `Asia/Shanghai`

#### 4.2.6 自动化质量保障几乎为空

- 现象：前端无 lint/test，后端只有一个空测试
- 影响：接口改动、重构、联调时很容易引入回归问题
- 位置：
  - `frontend/package.json`
  - `backend/src/test/java/com/cangqiong/takeaway/TakeawayApplicationTests.java`
- 建议：至少补充基础单元测试、接口层测试和前端 lint

### 4.3 低优先级问题

#### 4.3.1 Sass 写法已过时

- 现象：样式文件仍使用 `@import`
- 影响：当前不阻塞构建，但未来升级 Sass 时会产生兼容问题
- 位置：
  - `frontend/src/styles/global.scss`
  - `frontend/src/styles/element-plus.scss`
- 建议：迁移到 `@use` / `@forward`

#### 4.3.2 打包体积偏大

- 现象：前端构建已出现 chunk size warning，统计相关资源体积较大
- 影响：首屏加载与缓存效率可能较差
- 位置：`frontend/dist/assets/*`
- 建议：做图表按需加载、路由级拆包和依赖分析

#### 4.3.3 仓库产物治理不足

- 现象：当前目录里直接存在依赖与构建产物
- 影响：目录膨胀、协作混乱、压缩包增大、提交风险升高
- 当前已观察到：
  - `frontend/node_modules` 约 188 MB
  - `frontend/dist` 已存在
  - `backend/target` 已存在
- 建议：在仓库根目录建立统一 `.gitignore`，明确忽略规则

#### 4.3.4 项目根目录不是 Git 仓库

- 现象：根目录执行 Git 状态检查时提示不是 Git 仓库
- 影响：版本管理、审查、回滚、协作记录都不完整
- 建议：如果这是正式项目目录，应初始化或恢复 Git 仓库

## 6. 建议修复顺序

建议按以下顺序推进，而不是并行乱修：

1. 先修后端构建问题
   - 解决 Lombok 依赖
   - 确保 `mvn test` / `compile` 至少可跑通

2. 再统一数据库与 SQL 契约
   - 明确订单表名
   - 修复 `OrderMapper`、`StatisticsMapper` 与初始化脚本不一致问题

3. 再统一前后端接口契约
   - 先修订单页
   - 再修统计页和仪表盘
   - 明确状态字段、筛选参数、返回结构

4. 再补用户端真实接口
   - 先做认证
   - 再做购物车、下单、订单
   - 最后逐页去 mock 化

5. 最后补工程化收尾
   - README/docs 重写
   - lint/test
   - `.gitignore`
   - 截图与交付材料

## 8. 总结

这个项目已经具备较完整的“管理后台 + 用户端原型”外形，但距离“可稳定运行、可联调、可交付”的完整系统还有明显差距。

最核心的阻塞不是页面是否好看，而是下面三件事：

- 后端当前无法正常构建
- 订单与统计的前后端契约没有对齐
- 用户端仍以 mock 展示为主，没有真实后端支撑

如果只做展示或课程作业演示，目前已经能看出产品轮廓；如果要继续做成真正可交付项目，应优先按“构建修复 -> 数据库契约统一 -> 接口对齐 -> 用户端去 mock -> 工程化收尾”的顺序推进。
