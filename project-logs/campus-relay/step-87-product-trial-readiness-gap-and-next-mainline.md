# Step 87 - 产品级试运营前剩余差距清单 / go-no-go 评估

## 本轮目标

1. 基于 Step 86 已冻结试运营脚本线，评估当前项目距离“产品级试运营”还差什么。
2. 区分“本地答辩 / 交付 RC 已足够”和“产品级试运营仍需补齐”的边界。
3. 选择下一条最高价值主线。
4. 不改业务代码、不改 bridge、不改接口、不改路由、不改鉴权、不新增页面。

## 当前状态判断

当前项目已经达到“本地试运营 RC / 答辩交付”级别：

1. customer / courier / admin 主链路可演示。
2. 模拟支付、模拟退款、模拟打款口径已明确。
3. bridge 主线、展示 polish、媒体、地图、前端打包、脚本线都已冻结或收住。
4. H2/test profile + 本地 preflight + 样本校验 + 命令索引已能支撑本地演示和复跑。

但当前还不是“产品级上线”：

1. 没有真实部署环境。
2. 没有生产级密钥管理。
3. 没有 CI / 自动化回归入口。
4. 没有生产级监控、日志归档和备份策略。
5. 支付、退款、打款仍是模拟能力。
6. 腾讯地图只做单页 admin 只读点位预览，没有生产级配额、域名白名单和异常兜底策略。

## 产品级试运营差距矩阵

### 1. 环境与配置

当前状态：

1. `test profile + H2` 已适合本地试运营。
2. `dev / prod` 配置存在，但没有真实部署面验证。
3. 腾讯地图 key 通过 `frontend/.env.local` 本地配置，不提交真实 key。

差距：

1. 缺统一的环境变量样例和必填项说明。
2. 缺真实部署前的配置核对清单。
3. 缺 prod-like 启动验证。

建议优先级：P0。

### 2. 密钥与敏感信息

当前状态：

1. `frontend/.env.local` 未被 git 跟踪。
2. `preflight.ps1` 能检查地图 key 变量存在但不打印值。

差距：

1. 缺 `.env.example` 或等价说明，明确哪些 key 必填、哪些只用于本地。
2. 缺“不要把真实 key 提交到 repo”的统一开发者提醒。
3. 缺部署环境 secret 注入方式说明。

建议优先级：P0。

### 3. 部署与启动

当前状态：

1. 本地 backend / frontend 启动命令已整理。
2. H2 手动 reset 口径已整理。

差距：

1. 缺实际部署命令、部署目录结构、端口暴露和反向代理说明。
2. 缺部署后 smoke checklist。
3. 缺回滚版本说明。

建议优先级：P1。

### 4. CI / 自动化回归

当前状态：

1. 手工执行 backend compile、frontend build 和 preflight。
2. 没有 CI 流水线。

差距：

1. 缺自动跑 `.\mvnw.cmd -DskipTests compile` 与 `npm run build`。
2. 缺自动执行 sample validation。
3. 缺 PR / main push 后的基础质量门禁。

建议优先级：P1。

### 5. 监控、日志与问题定位

当前状态：

1. 本地日志靠控制台和 project-logs 留痕。
2. 没有集中日志或运行时监控。

差距：

1. 缺生产运行日志归档策略。
2. 缺错误告警、健康检查和基本指标。
3. 缺试运营问题记录模板。

建议优先级：P2。

### 6. 数据备份与恢复

当前状态：

1. 本地 H2 可通过重启重置。
2. MySQL schema / migration / seed 文件已维护。

差距：

1. 缺真实数据库备份策略。
2. 缺演示数据和试运营数据的恢复说明。
3. 缺迁移失败回滚口径。

建议优先级：P2。

### 7. 模拟资金链路

当前状态：

1. `docs/simulated-funds-boundary.md` 已明确 mock-pay、模拟退款、模拟打款和 settlement 审计边界。
2. 不接真实支付、真实退款、真实打款。

差距：

1. 如果进入真实商业试运营，需要补支付 / 退款 / 打款第三方对接方案。
2. 当前不应为了本地试运营提前接真实财务。

### 8. 外部服务与地图

当前状态：

1. 腾讯地图 JS SDK 已在 `/campus/courier-ops` 做单页 admin 点位预览。
2. 当前 key 本地配置，不提交 repo。

差距：

1. 缺域名白名单、配额策略和超额兜底说明。
2. 缺地图加载失败的生产级降级策略。
3. 不做路线规划、轨迹回放或实时调度。

建议优先级：P2。

## go / no-go 判断

### 是否继续做业务功能

结论：暂不优先。

原因：

1. 异常、售后、settlement 三条非 bridge 后端线已经具备可演示、可查询和最小处理能力。
2. 继续补业务能力会快速进入复杂化。
3. 当前更明显的短板是产品级运行边界，而不是功能数量。

### 是否进入产品级试运营前配置硬化

结论：建议进入。

原因：

1. 这是从本地 RC 迈向“可试运营”的最短路径。
2. 改动范围可以限定在配置样例、部署前清单和 secret hygiene，不触业务代码。
3. 能直接降低地图 key、环境变量、部署启动、运行检查的风险。
