# Step 155 — 双端移动原型结构对齐评估

## 本轮目标

对照用户端和兼职端两张移动端原型总览图，评估现有页面与原型的差距，输出差距清单和下一轮最小执行建议。本轮只做评估，不改代码。

## 原型图覆盖范围

### 用户端原型图（9 个页面/状态）

1. 登录页（手机号+验证码/密码双 tab）
2. 首页（欢迎语+发布新订单卡片+快捷入口）
3. 订单列表（状态筛选 tab：全部/待支付/配送中/已完成）
4. 订单详情（取餐码突出+路线+快递员信息+时间节点列表）
5. 等待确认（确认收货按钮+5 分钟自动确认倒计时）
6. 已评价（星级评分+评论）— 不实现
7. 异常订单（红色警告+异常详情+退款提示）— 退款语义不实现
8. 我的（个人中心：头像+姓名+学号+菜单）
9. 发布代送需求（表单）

### 兼职端原型图（11 个页面/状态）

1. 登录页（手机号+密码）
2. 工作台首页（今日统计：完成任务数/收入/平均配送时间）
3. 待接单列表（订单卡片：取件地/送达地/金额/操作按钮）
4. 订单详情（全屏页面：状态标签+路线+时间节点+操作按钮）
5. 取件确认（取餐码+备注输入）
6. 配送中/确认送达（路线+收件人+送达按钮）
7. 等待用户确认（已送达等待确认状态）
8. 异常订单（异常原因+处理状态）
9. 已完成订单（摘要+时间节点）
10. 个人资料（基础信息+审核状态）
11. 底部导航（工作台/我的/入驻）

## P0 / P1 / P2 优先级

### P0：影响主链路演示（必须修复）

1. **用户端订单状态时间轴** — 原型订单详情有完整时间节点列表（接单→取餐→送达），现有 OrderResult 只有时间字段没有可视化时间轴
2. **用户端"确认收货"功能** — 原型有"确认收货"按钮+5 分钟自动确认倒计时，现有 OrderResult 纯只读
3. **用户端订单列表 tab 筛选** — 原型用 tab 筛选（全部/待支付/配送中/已完成），现有用下拉 select

### P1：影响演示完整性

4. **兼职端工作台今日统计** — 原型有"完成任务数/收入/平均配送时间"，现有缺失
5. **用户端订单卡片设计** — 原型卡片有取餐码突出、路线、金额醒目，现有较简单
6. **用户端个人资料页** — 原型有"我的"页面（头像+菜单），需确认 `/user/profile` 内容
7. **兼职端任务卡片设计** — 原型卡片显示取件地→送达地+金额，现有较简单

### P2：可后续优化

8. 用户端登录页验证码 tab
9. 兼职端详情页改为全屏（当前 drawer 已功能完整）
10. 兼职端取件/送达/异常独立页（当前 drawer 内已覆盖）
11. 用户端首页简化（去掉旧兼容卡等冗余模块）

## 下一轮推荐执行包

### 候选方案

| 方案 | 内容 | 涉及文件 |
|------|------|----------|
| A | 用户端订单发布+模拟支付+订单结果三页联动视觉对齐 | CampusRelayOrders.vue, CampusOrderResult.vue |
| B | 兼职端工作台+任务详情+操作反馈视觉对齐 | CourierWorkbench.vue |
| C | 用户端入驻+token 申请视觉对齐 | CourierOnboarding.vue |
| D | 兼职端资料页+无 token/未审核/禁用状态视觉对齐 | Profile.vue, CourierWorkbench.vue |

### 推荐：方案 A

**理由：**

1. 覆盖 P0 全部 3 项（时间轴、确认收货、tab 筛选），影响最大
2. 涉及 2 个文件，改动范围可控
3. 用户端是演示主入口，视觉对齐收益最高
4. 原型图中用户端页面数量最多（9 个），差距最大
5. 方案 B/C/D 的 P0 项为 0，优先级低于 A

**方案 A 具体执行内容：**

1. CampusRelayOrders.vue：
   - 订单列表筛选从 el-select 改为 tab 筛选栏
   - 订单卡片增加取餐码突出、路线展示、金额醒目
   - 模拟支付按钮视觉优化

2. CampusOrderResult.vue：
   - 增加订单状态时间轴组件（接单→取餐→送达→完成）
   - 增加路线展示（取餐点→送达楼栋）
   - 确认收货按钮+倒计时提示（AWAITING_CONFIRMATION 状态时）
   - 各状态视觉区分增强

## 当前 bridge 结论

没有改 bridge。bridge 主线仍处于 Phase A no-op 冻结态。
