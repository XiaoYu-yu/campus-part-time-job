# Step 73 - 腾讯地图产品化线 go / no-go 与试点收口

## 本轮目标

1. 基于 Step 72 已完成的腾讯地图最小产品化试点，判断是否继续把地图承接扩到第二个既有页面。
2. 明确当前地图线是否继续推进，还是保留在 `CampusCourierOpsView.vue` 的单页试点状态。
3. 保持 bridge 主线、展示 polish 线、媒体线、非 bridge 后端三线和 settlement P3 线的既有冻结 / 收住口径。
4. 本轮不改业务代码、不改后端接口、不改鉴权、不改路由、不新增页面。

## Step 72 给本轮留下的起点

1. `CampusCourierOpsView.vue` 已接入腾讯地图 JS SDK 最小预览。
2. 地图预览复用现有配送员低频位置记录，不新增接口。
3. `/campus/courier-ops` 已通过运行态验证，能展示腾讯地图比例尺和版权信息。
4. 当前地图 key 未开启 WebService API，静态图路线已放弃，后续如继续地图线仍以 JS SDK 为主。
5. 真实地图 key 未提交到仓库，只通过本地未跟踪环境配置使用。

## 候选方向评估

### 方向 A：继续把地图承接扩到第二个既有页面

#### 可能承接页

1. `CourierWorkbench.vue`
   - 有订单详情和配送动作，但当前 courier 侧仍以最小承接为主。
   - 若加入地图，容易从“位置展示”滑向路线、导航、轨迹和调度。
2. `CampusOrderResult.vue`
   - customer 结果页是只读收尾页，当前没有必要展示地图。
   - 强行加入地图会增加 token、坐标、展示语义和隐私说明复杂度。
3. settlement / after-sale admin 页
   - 这些页面与地理位置关联弱。
   - 接地图收益低，容易形成“为了地图而地图”。

#### 收益

1. 可以扩大腾讯地图能力的可见范围。
2. 可在演示中说明地图不是孤立试点。

#### 风险

1. 容易把当前项目带入轨迹回放、路线规划、实时调度或导航产品线。
2. 第二页没有比 `CampusCourierOpsView.vue` 更自然的坐标上下文。
3. 会增加前端地图 key、SDK 生命周期、容器渲染和空态处理的维护面。
4. 当前项目刚进入产品化试点阶段，继续扩第二页的收益低于风险。

#### 结论

本轮不选择方向 A。

### 方向 B：保留当前 admin 地图试点并正式收住地图线

#### 收益

1. `CampusCourierOpsView.vue` 是当前最合适的地图承接点。
2. 已经验证真实地图 SDK 可用，能满足“从模拟数据走向真实地图能力”的产品化试点目标。
3. 单页试点足够支撑答辩、演示和后续产品化判断。
4. 不会把当前稳定交付状态重新拉入多页面地图改造。

#### 风险

1. 地图能力目前只在一个 admin 运营页可见。
2. courier / customer 侧暂时没有地图承接。
3. 如果后续要做真实调度、轨迹或路线规划，需要重新打开地图产品线。

#### 结论

本轮选择方向 B。

## 最终 go / no-go 结论

1. 对“扩到第二个既有页面”做 no-go。
2. 当前腾讯地图产品化线正式收住为：
   - 单页 admin 运营地图预览。
   - 只读点位展示。
   - 复用现有低频位置记录。
   - 不做轨迹、路线、调度、导航或地图写操作。
3. `CampusCourierOpsView.vue` 继续保留地图预览作为当前产品化试点结果。
4. `frontend/src/utils/tencentMap.js` 继续作为最小 SDK 加载工具保留。
5. 当前不新增第二个地图页面，不新增地图路由，不改后端接口。

## 恢复推进触发条件

只有出现以下真实条件之一，才建议重新打开地图产品化线：

1. 试运营明确要求 courier 或 customer 侧查看取送点地图。
2. 需要基于位置记录展示路线、距离或轨迹，并且后端已有足够数据支撑。
3. 腾讯地图 key 已正式开通更适合服务端能力的配额或 WebService API。
4. 当前 admin 地图试点在演示或试运营中被证明价值高，需要推广到第二个页面。
5. 有明确页面能自然承接地图，而不是为了扩大地图存在感强行接入。

重新打开前应先做 go / no-go，不应直接写代码。

## 明确没做

1. 没有修改 `CampusCourierOpsView.vue`。
2. 没有修改 `tencentMap.js`。
3. 没有新增第二个地图页面。
4. 没有新增路由。
5. 没有新增后端接口。
6. 没有改 bridge、鉴权、token 附着、`request.js` 或 `campus-*` API 运行时行为。
7. 没有提交真实腾讯地图 key。
8. 没有做轨迹回放、路线规划、实时调度、导航或位置写操作。

## 当前 bridge 结论

1. bridge 仍处于 `Phase A no-op` 冻结态。
2. `/api/campus/courier/profile` 与 `/api/campus/courier/review-status` 继续保留。
3. 本轮地图线收口不涉及 bridge 行为变化，也不构成 bridge 恢复推进触发条件。
4. 当前最终结论仍不是“bridge 已可删除”。

## 下一轮建议

1. Step 74 建议进入整体产品化下一阶段规划，而不是继续扩地图第二页。
2. 建议优先评估：
   - 模拟支付 / 试运营资金链路是否需要更明确的产品化封装。
   - 地图线是否保持冻结。
   - 当前所有冻结线是否已经足够支撑“试运营版”定义。
3. 若没有新的真实触发条件，不建议继续为地图、bridge、展示 polish、媒体或 settlement 追加新轮次。
