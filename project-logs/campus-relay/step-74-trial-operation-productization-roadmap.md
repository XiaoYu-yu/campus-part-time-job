# Step 74 - 试运营版产品化下一阶段规划

## 本轮目标

1. 基于 Step 73 已收住腾讯地图产品化试点，统一评估试运营版下一阶段最值得打开的产品化方向。
2. 不再机械追加地图页面、bridge 候选、展示 polish、媒体或 settlement 轮次。
3. 明确哪些主线继续冻结，哪些方向适合进入下一轮执行准备。
4. 本轮只做产品化规划，不改业务代码、不改接口、不改路由、不改鉴权、不新增页面。

## 当前稳定基线

1. 最小业务闭环已覆盖：
   - customer onboarding
   - admin 审核
   - courier token 申请
   - courier workbench
   - 接单、取餐、deliver、异常上报
   - customer confirm
   - completed 回读
   - customer 结果页
2. admin 运营侧已覆盖：
   - settlement 批次与单笔只读运营
   - 售后执行只读运营与执行历史
   - courier 异常 / 位置运营
   - 异常历史 / resolve 最小处理
   - settlement 批次操作审计
   - settlement 对账差异只读承接
3. 交付侧已覆盖：
   - 演示脚本
   - 截图清单
   - 录屏顺序
   - 媒体归档
4. 产品化试点侧已覆盖：
   - 腾讯地图 JS SDK 单页点位预览
   - 地图线 Step 73 已收住为单页 admin 只读试点

## 候选方向评估

### 方向 A：继续扩地图产品线

#### 可能内容

1. 把地图扩到 courier workbench。
2. 把地图扩到 customer 结果页。
3. 做轨迹、路线或调度面板。

#### 收益

1. 地图能力更显眼。
2. 更接近真实配送产品体验。

#### 风险

1. 容易从点位预览滑向轨迹、路线、调度、导航。
2. 当前只有低频位置记录，不足以支撑真实轨迹或路线能力。
3. 会引入更多地图 key、SDK、隐私和运行态维护成本。

#### 结论

当前不继续扩地图产品线。Step 72/73 的单页试点已经足够证明地图能力可接入。

### 方向 B：模拟支付 / 试运营资金链路产品化封装

#### 可能内容

1. 梳理模拟支付、模拟退款、模拟打款、settlement 的统一口径。
2. 明确哪些是模拟财务动作，哪些是只读审计。
3. 为试运营演示准备资金链路检查清单。

#### 收益

1. 能补齐“为什么不接真实支付也能试运营演示”的解释。
2. 能降低后续误把模拟支付当真实支付的风险。
3. 与当前 settlement、售后执行和对账差异线高度相关。

#### 风险

1. 如果直接写代码，容易误入真实支付、真实退款、真实打款。
2. 当前已有 settlement P3 和售后执行历史，继续扩动作收益下降。

#### 结论

值得进入下一阶段，但第一步应是“模拟资金链路产品化边界与运行手册”，不是直接写支付代码。

### 方向 C：试运营运行配置 / 环境 / 密钥 / 启动检查产品化

#### 可能内容

1. 统一整理本地试运营启动方式。
2. 明确 H2/test、MySQL/dev、前端 Vite 的配置差异。
3. 规范腾讯地图 key 等本地密钥的配置方式。
4. 补一份试运营 preflight：账号、样本数据、关键接口、地图 key、模拟资金状态。

#### 收益

1. 当前项目已经从“功能开发”进入“试运营可跑”阶段，运行配置是最直接的产品化短板。
2. Step 72 引入真实腾讯地图 key 后，更需要明确密钥不入库和本地环境配置方式。
3. 该方向不需要改业务代码，风险低。
4. 能直接提高交接、答辩、试运营复现质量。

#### 风险

1. 如果做成部署工程，会超出当前未部署前提。
2. 如果写得太抽象，不能真正帮助运行。

#### 结论

这是下一轮最高优先级。Step 75 建议进入“试运营运行配置与 preflight 手册”。

### 方向 D：继续扩 admin 运营处理动作

#### 可能内容

1. settlement review / withdraw / reconcile-difference resolve 前端动作。
2. 更多 admin 写操作或处理按钮。

#### 收益

1. 后台运营动作更完整。

#### 风险

1. 当前多条 admin 线已按“只读承接 + 最小处理”收住。
2. 继续加写动作会破坏当前冻结 / 维护态。
3. 易引出完整财务后台或工单系统。

#### 结论

当前不作为下一阶段主线。

### 方向 E：整体部署上线

#### 可能内容

1. 服务器部署。
2. Nginx / 网关配置。
3. MySQL 正式库。
4. 日志、监控、备份。

#### 收益

1. 接近真实上线。

#### 风险

1. owner 已确认项目此前没有部署发布。
2. 当前阶段用户目标仍是“模拟真实公司能力的试运营版”，不要求真实生产上线。
3. 直接上部署会拉入服务器、域名、证书、运维和安全问题。

#### 结论

暂不进入真实部署上线；可以在 Step 75 的运行手册中保留“未来部署准备项”。

## Step 74 最终结论

1. 地图线继续冻结为单页 admin 只读点位预览。
2. bridge 继续保持 `Phase A no-op` 冻结态。
3. 展示 polish 线继续保持冻结/维护态。
4. 媒体线继续收住。
5. 非 bridge 后端三线继续保持已收住状态。
6. 下一阶段最值得打开的是：
   - 试运营运行配置与 preflight 手册。
   - 模拟资金链路产品化边界说明。
7. 其中 Step 75 推荐先做“试运营运行配置与 preflight 手册”，因为它风险最低、收益最直接，并且能承接腾讯地图 key、账号、样本数据、模拟资金和本地启动验证。

## Step 75 推荐范围

### 推荐目标

1. 新增或整理一份可执行的试运营运行手册。
2. 覆盖：
   - backend 启动 profile
   - frontend 启动方式
   - H2/test 数据状态
   - MySQL/dev 注意事项
   - 腾讯地图 key 本地配置
   - 模拟支付 / 模拟退款 / 模拟打款口径
   - 关键页面和关键接口 preflight
   - 演示账号和样本订单检查
3. 不写业务代码。
4. 不提交任何真实密钥。
5. 不做真实部署。

### 明确不建议

1. 不继续扩地图第二页。
2. 不重开 bridge。
3. 不继续机械 polish。
4. 不新增第五个 admin 页。
5. 不接真实支付、真实退款、真实打款。

## 当前 bridge 结论

1. bridge 仍处于 `Phase A no-op` 冻结态。
2. `/api/campus/courier/profile` 与 `/api/campus/courier/review-status` 继续保留。
3. 本轮产品化规划不涉及 bridge 行为变化，也不构成 bridge 恢复推进触发条件。
4. 当前最终结论仍不是“bridge 已可删除”。

## 下一轮建议

Step 75 建议进入“试运营运行配置与 preflight 手册”：

1. 只做运行手册和检查清单。
2. 明确本地 key、profile、账号、样本数据、模拟资金链路和关键页面检查。
3. 不改业务代码、不改后端接口、不改前端路由、不新增页面。
